# 二十四节气展示页设计文档

## 1. 目标

构建一个基于 Next.js 的二十四节气展示页，以“首页即可完整浏览”为核心，提供兼具文化气质与信息密度的阅读体验。页面最终部署到 Vercel，首版使用内置静态数据完成全部展示与交互。

本设计的目标是：

- 用单页形式完整承载二十四节气内容，避免用户在多个页面之间跳转。
- 让用户可以按季节、月份快速筛选节气，并通过时间轴形成全年节律认知。
- 在视觉上体现“国风雅致”，但保持现代 Web 的清晰层级与可读性。
- 在工程上保持简单、稳定、可静态部署，便于快速上线与后续扩展。

### 2.1 本次实现范围

本次仅覆盖以下内容：

- 初始化 Next.js 项目
- 使用 App Router + TypeScript 完成首页

- 实现首页完整信息架构：
  - Hero 区

  - 时间轴导航

## 3. 产品定位与用户体验目标

### 3.1 产品定位

这是一个面向普通用户与传统文化兴趣用户的内容型展示页，重点不是工具效率，而是“浏览体验 + 知识传达 + 节律感知”。

页面应当同时满足两类使用路径：

- 快速浏览：用户进入首页后，通过视觉与卡片标题快速感知二十四节气全貌。
- 深度阅读：用户展开卡片后，可以阅读较完整的节气信息，包括简介、习俗、三候、诗句、养生或农事内容。

### 3.2 目标用户体验

用户进入页面后，应能在数秒内理解以下信息：

- 这是一个关于二十四节气的页面
- 可以按四季或月份筛选
- 可以通过时间轴在全年节律中跳转
- 每张卡片都可以展开查看更丰富的信息

体验原则如下：

- 首屏明确：标题、氛围、页面用途一眼可知
- 浏览顺滑：筛选、导航、展开无需思考
- 信息丰富但不混乱：内容多，但层级清晰
- 移动端可用：在手机上仍可完成浏览、筛选和展开
- 无需等待：首版静态数据优先保证页面直出和响应性

## 4. 信息架构

首页采用单页分层信息架构，按“认知建立 -> 条件筛选 -> 时间导航 -> 内容浏览 -> 补充说明”展开。

### 4.1 页面结构总览

1. Hero 区
2. 筛选区
3. 时间轴导航
4. 节气卡片网格
5. 页尾说明

### 4.2 Hero 区

用途：

- 建立页面主题
- 传达东方节律与文化气质
- 给出简洁导语，说明可浏览、筛选、查看详解

建议内容：

- 主标题：二十四节气
- 副标题/说明：围绕“顺时而生、观岁时流转”之类的语义
- 可选短文案：说明可通过季节、月份与时间轴浏览全年节气

Hero 不承担复杂操作，重点是建立氛围与认知入口。

### 4.3 筛选区

包含：

- 四季筛选
- 月份筛选
- 重置按钮

作用：

- 让用户按认知最自然的方式缩小内容范围
- 作为主要交互入口之一，与时间轴形成互补

筛选区应固定为轻量交互区，不承担说明型大段文本。

### 4.4 时间轴导航

用途：

- 将二十四节气放回“全年时间序列”中理解
- 弥补卡片网格在时序感上的弱点
- 帮助用户快速跳转到某个节气或某段时间

表现形式：

- 轻量时间轴
- 强调顺序感而非复杂图表感
- 可按全年顺序展示 24 个节气节点，当前筛选结果中的节点高亮，其余弱化或保留可点击状态

### 4.5 节气卡片网格

用途：

- 作为主内容区承载全部节气卡片
- 支持浏览和展开阅读

卡片默认展示精简信息，展开后展示详细内容。网格应适配不同屏宽：

- 移动端：单列或紧凑双列策略中优先保证可读性
- 平板端：双列
- 桌面端：多列网格

### 4.6 卡片展开内容

展开后展示的信息模块：

- 简介
- 习俗
- 三候
- 诗句
- 养生/农事

这些内容应作为结构化块显示，而不是混成一段长文本，便于扫描阅读。

### 4.7 页尾说明

用途：

- 说明当前版本数据为内置静态内容
- 简述页面设计意图
- 为未来扩展留出说明空间
- 可加入简短版权/说明信息

## 5. 视觉方向

### 5.1 总体风格

视觉方向为“国风雅致”，关键词如下：

- 克制
- 温润
- 留白
- 节律感
- 书卷气
- 现代排版下的东方气质

不是仿古页面，也不是浓重装饰风。应避免：

- 过度拟物
- 高饱和大面积纯色
- 复杂纹样堆叠
- 过重插画导致内容喧宾夺主

### 5.2 色彩建议

建议以低饱和、自然系为主，例如：

- 米白、宣纸白作为背景基调
- 墨色、黛青用于正文与层级文字
- 松花绿、竹青、秋棕、梅红等作为季节性点缀
- 金色或浅赭色用于弱强调，不应大面积使用

色彩应服务于：

- 季节区分
- 状态高亮
- 内容层级

### 5.3 字体与排版建议

优先保证 Web 端兼容与可读性，首版不依赖特殊字体资源。可通过以下方式建立气质：

- 标题使用更疏朗的字号与字重
- 正文采用清晰、舒适的现代字体栈
- 控制行长与段落间距
- 使用细边框、浅底色、分组标题建立秩序感

### 5.4 装饰元素建议

可适度加入轻量装饰，但不依赖装饰完成设计：

- 细线分隔
- 淡纹理背景
- 小型节令标识
- 季节色彩标签
- 圆角卡片与浅阴影

装饰必须服从可读性与性能。

## 6. 技术方案

### 6.1 技术选型

已确认方案：

- Next.js
- App Router
- TypeScript
- Tailwind CSS
- 静态数据文件
- React 本地状态
- 筛选状态同步到 URL query
- 部署到 Vercel

该方案适合当前目标，原因如下：

- Next.js 便于后续继续扩展页面结构和部署能力
- App Router 适合当前项目目录组织与后续升级
- TypeScript 有助于数据结构和组件边界清晰化
- Tailwind CSS 能快速建立统一视觉系统
- 静态数据适合首版快速交付与稳定部署
- Vercel 与 Next.js 配合成熟，部署成本低

### 6.2 渲染策略

首版页面基于静态数据构建，优先采用静态输出友好的实现方式。页面内容在构建期即可确定，不依赖运行时接口请求。

预期效果：

- 首屏快
- 部署简单
- 内容稳定
- 搜索引擎可直接获取主要文本内容

### 6.3 项目结构建议

以下为建议的目录组织，不假设仓库内已存在脚手架，仅作为实现依据：

```txt
app/
  page.tsx
components/
  superpowers/
    solar-terms/
      hero.tsx
      filter-bar.tsx
      timeline-nav.tsx
      solar-term-card.tsx
      solar-term-grid.tsx
data/
  solar-terms.ts
types/
  solar-term.ts
lib/
  solar-terms/
    filters.ts
    query.ts
```

说明：

- `app/page.tsx` 作为首页入口，负责组装页面
- `components/superpowers/solar-terms/` 存放页面专属组件
- `data/solar-terms.ts` 存放静态数据
- `types/solar-term.ts` 定义核心数据类型
- `lib/solar-terms/` 放筛选和 URL 解析等纯函数逻辑

## 7. 数据模型草案

### 7.1 数据设计原则

静态数据需要同时满足：

- 列表展示
- 筛选判断
- 时间轴排序
- 展开详情渲染
- 后续迁移到 CMS/API 时具备清晰字段映射

因此数据应避免“全都塞进 description”式的松散结构，应采用结构化字段。

### 7.2 单个节气数据建议结构

```ts
type Season = 'spring' | 'summer' | 'autumn' | 'winter'

type SolarTerm = {
  id: string
  name: string
  pinyin?: string
  season: Season
  month: number
  order: number
  approximateDate: string
  summary: string
  customs: string[]
  pentads: string[]
  poem?: {
    title?: string
    author?: string
    content: string
  }
  healthOrFarming: string[]
}
```

### 7.3 字段说明

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `string` | 唯一标识，建议使用稳定英文 slug，如 `lichun` |
| `name` | `string` | 节气中文名 |
| `pinyin` | `string` | 可选，便于视觉补充或后续搜索 |
| `season` | `Season` | 所属季节，用于筛选与视觉标识 |
| `month` | `number` | 对应月份，用于筛选 |
| `order` | `number` | 1-24 的全年顺序，用于时间轴与排序 |
| `approximateDate` | `string` | 节气大致公历时间描述，如“2月3日-5日” |
| `summary` | `string` | 节气简介 |
| `customs` | `string[]` | 习俗列表 |
| `pentads` | `string[]` | 三候列表，通常 3 条 |
| `poem` | object | 诗句信息，可选 |
| `healthOrFarming` | `string[]` | 养生建议或农事提示 |

### 7.4 数据组织建议

静态数据文件建议导出：

- 完整数组：用于页面渲染
- 季节/月份常量：用于筛选 UI
- 可能的映射表：用于 query 解析与显示文案统一

例如：

```ts
export const solarTerms: SolarTerm[] = [...]
export const seasonOptions = [...]
export const monthOptions = [...]
```

## 8. 组件拆分设计

### 8.1 页面入口：`app/page.tsx`

职责：

- 解析 URL query
- 初始化筛选状态
- 根据状态计算筛选结果
- 组织 Hero、筛选栏、时间轴、卡片网格、页尾说明
- 将状态和回调下发给子组件

不建议在该文件中堆叠过多渲染细节。页面入口应保持“页面编排层”角色。

### 8.2 Hero 组件

建议命名：`Hero`

职责：

- 展示标题、副标题、导语
- 承担视觉氛围建立
- 不管理业务状态

输入：

- 静态文案即可，必要时可通过 props 传入标题与说明

### 8.3 筛选栏组件

建议命名：`FilterBar`

职责：

- 渲染四季筛选按钮
- 渲染月份筛选按钮/下拉
- 提供重置操作
- 响应当前已选状态

输入建议：

- `selectedSeason`
- `selectedMonth`
- `onSeasonChange`
- `onMonthChange`
- `onReset`

输出仅通过回调完成，不直接处理 URL。

### 8.4 时间轴导航组件

建议命名：`TimelineNav`

职责：

- 按全年顺序展示 24 个节气节点
- 标识当前筛选结果
- 支持点击定位到某节气卡片

输入建议：

- `items`
- `activeFilters`
- `onJumpToTerm` 或通过锚点跳转实现

若采用 DOM 锚点跳转，组件只负责生成链接；若采用编程滚动，则由父层协调。

### 8.5 节气卡片组件

建议命名：`SolarTermCard`

职责：

- 展示单个节气的概览
- 处理卡片展开与收起
- 在展开态展示详细信息模块

输入建议：

- `item`
- `expanded`
- `onToggle`

为保证状态可控，展开状态建议由父层统一管理，而不是每张卡片内部自管。

### 8.6 节气卡片网格组件

建议命名：`SolarTermGrid`

职责：

- 接收筛选后的节气列表
- 统一渲染网格布局
- 管理“当前展开的是哪一张卡片”或将该状态从父层透传
- 在空结果时展示空状态提示

输入建议：

- `items`
- `expandedId`
- `onToggleExpand`

## 9. 状态与 URL 设计

### 9.1 状态范围

页面核心状态包括：

- 当前季节筛选
- 当前月份筛选
- 当前展开卡片
- 当前筛选结果

其中：

- 筛选状态需要同步到 URL query
- 展开状态不强制写入 URL，首版保留在本地状态即可

### 9.2 状态模型建议

```ts
type FilterState = {
  season?: 'spring' | 'summer' | 'autumn' | 'winter'
  month?: number
}
```

额外本地状态：

```ts
type ExpandedState = {
  expandedId?: string
}
```

### 9.3 URL Query 设计

建议使用以下 query 参数：

- `season`: 季节
- `month`: 月份

示例：

- `/`
- `/?season=spring`
- `/?month=3`
- `/?season=autumn&month=8`

设计原则：

- 参数名短小且语义清晰
- 空状态时不保留无意义参数
- 参数非法时自动忽略并回退到默认展示

### 9.4 URL 与状态同步原则

- 初次进入页面时，从 URL query 解析筛选状态
- 用户点击筛选控件后，更新本地状态并同步更新 URL
- 重置操作应清空 query，回到无筛选状态
- 不要求页面刷新后保留展开中的卡片状态

### 9.5 筛选逻辑约定

筛选逻辑为“交集”关系：

- 只选季节：显示该季节所有节气
- 只选月份：显示该月份相关节气
- 同时选季节和月份：显示同时满足两者的节气
- 无筛选：显示全部 24 个节气

需考虑的边界：

- 若季节与月份组合后为空，展示明确空状态
- URL 中月份不是 1-12 时视为无效
- URL 中季节不在枚举值内时视为无效

## 10. 交互细节

### 10.1 季节筛选

- 点击某个季节后，进入该季节筛选状态
- 再次点击同一季节，可取消该季节筛选，回到未选择状态
- 季节按钮需有明确选中态、悬停态、键盘聚焦态

### 10.2 月份筛选

- 月份作为更细粒度筛选入口
- 可采用横向按钮组、折行按钮组或紧凑下拉，但首版建议按钮组优先，直观可见
- 点击已选月份可取消选择

### 10.3 重置操作

- 一键清空季节和月份筛选
- URL 回到无 query 状态
- 节气网格恢复全部内容

### 10.4 时间轴导航

时间轴应满足：

- 能清晰体现全年顺序
- 节点可点击
- 点击后跳转到对应节气卡片位置
- 当前筛选结果中的节点可高亮
- 不在筛选结果中的节点可弱化，但不必完全隐藏，以保留全年结构感

若目标节气当前因筛选未显示，有两种策略：

1. 保持不可跳转并视觉弱化
2. 点击时自动清除冲突筛选并跳转

首版建议采用更稳定、认知更直接的策略：

- 时间轴点击仅对当前可见项生效
- 不可见项弱化且不可触发跳转

### 10.5 卡片展开

- 每张卡片默认折叠展示概览
- 点击“展开”按钮或卡片主要操作区域后展开详情
- 再次点击可收起
- 首版建议同一时刻仅展开一张卡片，减少页面高度失控和阅读干扰

展开后内容顺序建议为：

1. 简介
2. 习俗
3. 三候
4. 诗句
5. 养生/农事

### 10.6 空状态

当筛选结果为空时，应显示友好提示，例如：

- 当前筛选下没有匹配的节气
- 可尝试切换季节、月份或重置筛选

空状态不应只是空白页面。

### 10.7 可访问性

首版仍应满足基础可访问性要求：

- 按钮使用语义化元素
- 展开/收起按钮暴露状态
- 时间轴可键盘访问
- 颜色不作为唯一状态区分手段
- 文本对比度满足基本阅读要求

## 11. 页面内容组织建议

### 11.1 卡片折叠态内容

建议展示：

- 节气名称
- 拼音或简短副标（可选）
- 所属季节 / 月份
- 大致日期
- 一句话简介

目标是让用户在不展开时也能完成快速浏览。

### 11.2 卡片展开态内容

#### 简介
用于解释该节气的核心含义和时令特征。

#### 习俗
使用列表展示，便于扫描阅读。

#### 三候
固定为三条列表，强调节令观察体系。

#### 诗句
单独分组显示，形成文化气质亮点。

#### 养生/农事
根据节气属性给出实用导向内容。若某节气更适合农事表述，可统一放入该模块，不必强行分裂为两个模块。

## 12. 工程实现约定

### 12.1 组件边界

- 展示组件尽量纯粹，避免混入 URL 解析逻辑
- URL query 的读写集中在页面层或专用 hook/工具函数中
- 筛选与排序逻辑放入纯函数，便于测试和复用

### 12.2 纯函数建议

建议拆出以下工具函数：

- `parseFiltersFromSearchParams`
- `serializeFiltersToQuery`
- `filterSolarTerms`
- `getSolarTermAnchorId`

这样可以减少 UI 组件中的条件分支。

### 12.3 锚点与滚动定位

为时间轴导航与卡片联动，建议为每个节气卡片生成稳定锚点：

```txt
#term-lichun
#term-yushui
...
```

这样可以：

- 简化跳转实现
- 便于未来分享特定节气位置
- 降低编程式滚动的复杂度

### 12.4 样式组织建议

Tailwind CSS 使用上建议：

- 颜色、间距、圆角、阴影尽量复用统一模式
- 避免在组件中出现大量难以维护的超长 class 组合
- 可通过适度提取组合样式或保持结构分层清晰来控制复杂度

## 16. 风险与控制点

### 16.1 信息密度过高导致首屏压迫

风险：

- 内容丰富可能导致页面显得沉重

控制方式：

- 通过折叠态卡片控制首屏信息量
- 通过清晰留白和分组标题降低阅读压力
- 将详细内容放在展开态中承载

### 16.2 筛选与时间轴联动增加复杂度

风险：

- 用户可能不理解时间轴为何有些节点不可跳转

控制方式：

- 明确高亮与弱化逻辑
- 保持当前可见项可跳转、不可见项弱化
- 避免首版引入“自动改写筛选条件”的复杂行为

### 16.3 静态数据维护成本

风险：

- 内容修改需要直接改代码

控制方式：

- 保持数据结构清晰
- 将文本集中在单一数据文件
- 为未来迁移到 CMS 保留字段语义一致性

## 18. 实施结论

本方案以“单页完整浏览”为核心，将二十四节气内容组织为一个具有文化气质、信息丰富且交互清晰的首页。技术上采用 Next.js App Router、TypeScript、Tailwind CSS 与静态数据方案，确保首版能够快速落地、稳定运行并顺利部署到 Vercel。

本设计的关键判断如下：

- 首页卡片浏览是首版最适合的承载形式
- 季节筛选、月份筛选与轻量时间轴共同构成高效而自然的浏览方式
- 静态数据是当前阶段最合理的数据策略
- URL query 同步筛选状态，兼顾可分享性与可恢复性
- 将复杂度控制在单页与本地状态范围内，有利于快速交付并为后续演进保留空间

该文档可直接作为后续实现、组件拆分、数据建模与验收的依据。

# 二十四节气插画生成阶段设计规格文档

## 1. 背景与目标

当前项目已具备以下基础：

- 已有二十四节气长卷首页，作为主要浏览与叙事入口。
- 已有 `SolarTerm` 数据结构。
- 已有面向概念描述的 `imageBrief` 字段。
- 已有用于承载插画的画境区：`solar-term-panel__painting`。
- 首页当前结构与视觉方向已确定，不希望因插画生成阶段而打断现有长卷体验。

在此基础上，插画生成阶段的目标不是立即重构首页，而是先建立一套可执行、可预览、可交付的插画规格体系，用于支持后续 AI 成图、筛选、回填与页面落位。

本阶段目标如下：

1. 新增一个独立的“插画规格预览页”，不改动当前首页长卷结构。
2. 通过 3 个代表节气样本，验证整套插画规格是否清晰、稳定、可用于批量生产。
3. 将预览页同时定义为：
   - 视觉预览页：用于快速查看插画方向是否统一；
   - 生产说明页：用于指导 AI 生成、人工筛选、后续资源交付。
4. 在数据层面保留 `imageBrief`，并新增执行层字段（建议命名为 `imageSpec`），建立“创意描述”与“生产规范”分层。
5. 为未来将真实插画插入 `SolarTermPanel` 左侧 `solar-term-panel__painting` 区域做好规格前置。

一句话概括：本阶段要先把“怎么出图、出什么图、如何放进页面”定义清楚，再进入批量生成与集成阶段。

---

## 2. 范围与非目标

### 2.1 本阶段范围

本阶段文档覆盖以下内容：

- 独立插画规格预览页的定位与信息架构
- 3 个代表节气的样本展示策略
- 插画输出比例、推荐尺寸、安全区与构图约束
- 插画未来在 `solar-term-panel__painting` 中的插入原则
- `imageBrief` 与 `imageSpec` 的数据分层建议
- 预览页应承载的说明模块
- 资源命名、交付格式、成图管理规范
- 风险与验收标准

### 2.2 明确非目标

以下内容不属于本阶段目标：

- 不改动当前首页长卷结构与叙事顺序
- 不在本阶段完成全部 24 张节气插画
- 不在本阶段确定最终 AI 模型、供应商或工作流平台
- 不在本阶段实现复杂的图片裁切编辑器或人工标注系统
- 不在本阶段替换、删除或废弃 `imageBrief`
- 不以预览页替代首页，预览页仅作为独立说明与校验页面
- 不在本阶段定义品牌级整站视觉系统重构

## 3. 预览页信息架构

预览页应被定义为“插画生产与页面适配的统一说明入口”。其信息架构应同时满足设计查看、生成执行、前端接入三类需求。

建议采用以下自上而下结构：

### 3.1 页面级结构

1. 页面标题区
2. 项目说明区
3. 全局插画规格区
4. Safe Area 规则区
5. 插入位置设计区
6. 代表节气样本区（立春 / 清明 / 大雪）
7. 数据模型说明区
8. 文件命名与交付规范区
9. 实现建议与风险区
10. 验收标准区

### 3.2 页面阅读路径

建议阅读路径为：

- 先理解“为什么要做这页”
- 再理解“图片必须长什么样”
- 再理解“为什么选择这 3 个节气”
- 再查看“每个样本如何定义”
- 最后落到“如何命名、如何交付、如何验收”

这样可以避免预览页沦为纯展示页，也避免它变成只有规则、没有视觉感知的说明文档。

### 3.3 单个节气样本卡片建议结构

每个代表节气的展示模块建议包含：

- 节气名称
- 节气英文或拼音标识（如已有则可展示）
- 简短选择理由
- 当前 `imageBrief`
- 建议新增的 `imageSpec`
- 构图说明
- Safe Area 说明
- 页面插入预期效果说明
- 成图禁忌项

这能让预览页同时服务于内容策划、AI 出图和前端接入。

## 4. 图片规格

### 4.1 主规格

本项目节气插画的主规格定义为：

- 比例：`4:5`
- 推荐尺寸：`1536 × 1920`

该规格作为后续 AI 成图的主输出标准，适用于当前阶段的预览验证与后续页面接入。

### 4.2 选择 4:5 的原因

选择 `4:5` 作为主规格，原因如下：

1. 相比正方形更适合表现人物、植物、天气与地景的纵向叙事。
2. 相比过长竖图更容易控制主体集中度，便于嵌入页面左侧画境区。
3. 对 AI 生成而言，`1536 × 1920` 兼顾清晰度、可控性与常见模型支持度。
4. 在前端页面中，4:5 更利于形成稳定、统一的节气视觉节奏。
5. 后续若需为社交分享、详情页或局部裁切输出，也有足够的二次适配空间。

### 4.3 输出要求

所有正式进入候选池的插画资源应满足：

- 保持 4:5 比例
- 基准分辨率不低于 `1536 × 1920`
- 画面主体完整，不因边缘裁切而损伤主要语义
- 具备清晰层次：前景 / 主体 / 环境 / 气候氛围
- 不依赖大段文字或版式信息传递含义
- 即使脱离文案，也能传递节气气质

### 4.4 适配原则

虽然输出主规格固定为 4:5，但使用时需接受以下事实：

- 页面实际显示区域未必与源图完全等比
- 不同屏幕宽度下可能存在缩放
- 局部裁切可能发生，但必须在 Safe Area 规则内进行
- 前端不应依赖“恰好完整显示整张图”作为唯一正确展示方式

因此，插画规格必须从源头支持“可轻度裁切、主体不丢失”。

## 5. Safe Area 规则

Safe Area 是本项目插画可生产、可适配、可稳定落位的核心规则。所有 AI 成图与人工筛选都应优先遵守。

## 5.1 主体位置原则

主体应放在画面中部区域，不应顶天立地，也不应严重偏边。

建议理解为：

- 主体视觉重心位于中间 60% 左右的核心区域
- 主体轮廓与重要细节应尽量集中在中心带
- 允许一定方向性偏移，但不应把关键人物、树冠、建筑主结构、飞鸟主群等压到边缘

### 5.2 顶部留白

顶部建议预留 `10% - 12%` 的空间。

目的：

- 给页面适配、呼吸感和气候氛围留空间
- 避免头部、枝叶尖端、山体顶点过于贴边
- 为后续可能出现的光感、天气层、雾气层留余地

不建议出现的问题：

- 主体头部或顶部关键景物紧贴画面上边
- 重要元素被顶部裁切后语义受损
- 顶部完全塞满，导致画面压迫感过强

### 5.3 底部留白

底部建议预留 `12% - 15%` 的空间。

目的：

- 预留页面嵌入后的下沿呼吸区
- 防止地面、衣摆、花丛、雪地反光等重要内容贴底
- 降低页面容器裁切时切掉关键细节的风险

不建议出现的问题：

- 人物脚部或地景核心结构贴底
- 底部装饰层过重，导致后续显示时拥挤
- 重要叙事元素落在最底部边缘

### 5.4 左右边界规则

左右两侧避免关键主体贴边。

具体要求：

- 人物面部、手部、主植物、主建筑、主飞行动物不应贴左右边缘
- 左右边缘可承载环境气氛、枝叶延展、云气、风雪、光晕，但不应承载唯一核心信息
- 如果画面采用偏左或偏右构图，另一侧必须有足够视觉回弹空间

### 5.5 建议的安全区理解方式

可将整张图拆成三层区域理解：

- 外围风险区：上下左右边缘约 8% - 12% 范围，只放辅助环境信息
- 中层过渡区：可放次级叙事元素
- 中心主叙事区：放主体、主动作、主情绪、主季节符号

### 5.6 适用对象

Safe Area 规则适用于：

- AI 出图 prompt 约束
- 候选图筛选
- 人工修图与二次裁切
- 前端接入前资源验收
- 未来批量生成的全 24 节气插画

## 6. 插入位置设计

### 6.1 现有插入位置

真实插画未来将继续插入当前 `SolarTermPanel` 左侧的 `solar-term-panel__painting` 区域。

这是本阶段必须坚持的前提，不因新增预览页而改变首页结构。

### 6.2 插入位置的设计含义

`solar-term-panel__painting` 并不只是“放一张图”的容器，它承担的是：

- 节气氛围建立
- 面板内容的视觉引子
- 文本内容的情绪前置
- 长卷节奏中的停顿与观赏点

因此，插画不是装饰性配图，而是节气面板的重要组成部分。

### 6.3 对插画源图的反向约束

由于插画会进入左侧画境区，源图设计需要反向满足以下条件：

1. 单张图必须具备独立观赏性。
2. 在较窄容器中显示时，主体仍应成立。
3. 左侧图像不应依赖右侧文字补全才看得懂。
4. 画面视觉冲击不能强到压过节气标题与正文。
5. 图像边缘必须足够干净，避免嵌入时显得拥堵。

### 6.4 页面落位建议

后续实际插入时建议遵守：

- 优先完整显示主体，不强求完整显示全部背景
- 如有裁切，优先裁掉外围环境层，不裁主体层
- 不在图上叠加大面积强对比文案
- 如需悬浮层、遮罩或渐变，应保证不破坏画面主情绪
- 对 3 个代表节气的样本可先验证实际嵌入感，再推广到全部节气

## 7. 3 个代表节气选择理由

本阶段仅展示 3 个代表节气：立春、清明、大雪。该选择不是随机抽样，而是为了用最少样本覆盖最多的节气视觉类型。

## 7.1 立春

立春代表“新生、转折、萌动”。

适合作为样本的原因：

- 能验证“早春初醒”类气质表达
- 适合测试嫩绿、微冷、晨光、风感等细腻氛围
- 可覆盖植物萌发、人物迎春、冰雪初融等视觉语汇
- 能帮助建立全年节气插画的起点基调

它对应的是“生发型画面”的标准样本。

## 7.2 清明

清明代表“湿润、明净、轻哀、行旅、烟雨”。

适合作为样本的原因：

- 兼具自然气候与文化情绪双重特征
- 适合测试雨雾、柳色、水气、行路感等中景叙事
- 对 AI 成图而言，清明能很好检验“含蓄氛围”是否能稳定控制
- 有助于建立非强烈色彩条件下的画面层次方法

它对应的是“情绪型画面”的标准样本。

## 7.3 大雪

大雪代表“严冬、覆盖、静谧、厚重、寒意”。

适合作为样本的原因：

- 能测试低温、高白度、雪景层次是否可控
- 可验证大面积留白下如何保持主体稳定
- 有助于建立冬季节气在冷色环境中的统一标准
- 与立春、清明形成明显季节对比，覆盖跨度大

它对应的是“极端季候型画面”的标准样本。

## 7.4 三者组合的代表性

立春、清明、大雪组合后，可以覆盖：

- 春初生发
- 春中湿润
- 冬季凝结

它们共同覆盖了：

- 明亮但不炫目的自然色调
- 雨雾与空气感
- 白雪与冷冽环境
- 人物/植物/地景三类常见构图语境
- 从轻盈到厚重的节气情绪跨度

因此，用这 3 个样本建立首版规格，具有较高代表性与推广价值。

## 8. 数据模型建议：`imageBrief` / `imageSpec` 分层

## 8.1 分层原则

建议明确区分两层信息：

### `imageBrief`

用于表达创意层、概念层、叙事层的描述。

它回答的问题是：

- 这张图想表达什么？
- 这个节气的气质是什么？
- 应该出现怎样的意象、氛围和情绪？

### `imageSpec`

用于表达执行层、生产层、交付层的规范。

它回答的问题是：

- 图应该按什么比例生成？
- 主体放在哪里？
- 哪些元素必须避开边缘？
- 最终资源如何命名、交付、验收？

这种分层能保证创意与执行既关联又不混淆。

## 8.2 为什么保留 `imageBrief`

保留 `imageBrief` 是必要的，原因如下：

1. 它已经存在，具备延续性。
2. 它适合描述节气意境，不应被技术字段替代。
3. 它更适合服务策划、编辑、概念设计。
4. 它可以长期作为生成 prompt 的高层输入。
5. 即使未来更换模型或生成工具，`imageBrief` 仍然有效。

## 8.3 为什么新增 `imageSpec`

新增 `imageSpec` 同样必要，原因如下：

1. AI 生成阶段需要更明确的硬约束。
2. 页面接入阶段需要可执行的落位信息。
3. 资源交付阶段需要统一命名、尺寸、格式标准。
4. 后续批量化时，执行规范必须结构化，不能只靠自然语言描述。
5. 可减少不同人理解同一 `imageBrief` 时的偏差。

## 8.4 建议字段结构

以下为建议结构示意，字段名可后续按项目风格细化，但分层思路应保持不变：

```ts
type ImageBrief = {
  summary: string
  mood: string[]
  imagery: string[]
  seasonalSignals: string[]
}

type ImageSpec = {
  aspectRatio: "4:5"
  recommendedSize: {
    width: 1536
    height: 1920
  }
  safeArea: {
    top: "10%-12%"
    bottom: "12%-15%"
    leftRight: "avoid-key-subject-near-edge"
    subjectPlacement: "center-middle"
  }
  composition: {
    subjectFocus: string
    foreground?: string
    background?: string
    depth?: string
  }
  insertionTarget: {
    component: "SolarTermPanel"
    slot: "solar-term-panel__painting"
  }
  productionNotes: string[]
  negativeConstraints: string[]
  delivery: {
    fileFormat: string[]
    namingPattern: string
  }
}
```

## 8.5 字段职责边界

建议边界如下：

| 字段 | 作用 | 是否偏创意 | 是否偏执行 |
|---|---|---:|---:|
| `imageBrief.summary` | 概括画面意境 | 是 | 否 |
| `imageBrief.mood` | 情绪关键词 | 是 | 否 |
| `imageBrief.imagery` | 意象元素 | 是 | 否 |
| `imageSpec.aspectRatio` | 输出比例 | 否 | 是 |
| `imageSpec.recommendedSize` | 输出尺寸 | 否 | 是 |
| `imageSpec.safeArea` | 安全区约束 | 否 | 是 |
| `imageSpec.composition` | 构图执行说明 | 中 | 是 |
| `imageSpec.productionNotes` | 出图注意事项 | 否 | 是 |
| `imageSpec.negativeConstraints` | 禁忌项 | 否 | 是 |
| `imageSpec.delivery` | 交付规范 | 否 | 是 |

## 8.6 推荐使用方式

后续生产建议采用以下顺序：

1. 先读 `imageBrief`
2. 再读 `imageSpec`
3. 基于两者合成 AI prompt 或人工插画说明
4. 成图后按 `imageSpec` 验收
5. 验收通过后再接入 `solar-term-panel__painting`

这样能形成从概念到落地的闭环。

## 9. 预览页内容模块

预览页不应只是三张图并排展示，而应是完整的“规格说明 + 样本验证”页面。建议至少包含以下模块。

## 9.1 页面总说明模块

用于说明：

- 本页不是首页替代
- 本页是插画生成阶段的预览与生产说明入口
- 当前只展示 3 个代表节气样本
- 通过样本验证后再推广到全部 24 节气

## 9.2 全局规格模块

用于集中展示所有节气共用的硬规则：

- 比例：4:5
- 推荐尺寸：1536 × 1920
- 使用目标：AI 成图主规格
- 适配场景：后续插入 `solar-term-panel__painting`

该模块应尽量简洁、可扫读。

## 9.3 Safe Area 模块

用于把“主体居中、顶部留白、底部留白、左右避边”等规则明确可视化表达。

即使先不做复杂示意图，文字也应清晰到足以直接执行。

## 9.4 插入位置说明模块

用于说明真实插画未来如何回到现有页面体系：

- 插入位置在哪里
- 为什么需要独立预览页
- 预览页如何服务首页集成
- 页面嵌入时优先保护什么内容

## 9.5 代表节气样本模块

分别展示：

- 立春
- 清明
- 大雪

每个样本建议包含以下子模块：

1. 节气概述
2. 选择理由
3. `imageBrief`
4. `imageSpec`
5. 建议构图
6. Safe Area 提示
7. 禁忌项
8. 页面嵌入预期

## 9.6 数据模型说明模块

用于解释为什么要做 `imageBrief` / `imageSpec` 分层，以及后续如何让这两类字段协同工作。

## 9.7 交付规范模块

用于统一：

- 文件命名
- 文件格式
- 尺寸
- 版本管理
- 候选图与最终图区别

## 9.8 验收模块

用于给出可直接执行的验收清单，避免生成后只能凭主观判断。

## 9A. 统一风格基线

为避免立春、清明、大雪在验证阶段各自发展成不同画风，需补充一组可直接执行的统一风格基线。后续全部 24 节气若进入批量生产，也应默认继承本章节。

### 9A.1 整体视觉方向

本项目节气插画统一采用以下方向：

- 以“东方季候感 + 现代插画叙事”为主，不走纯国画复刻，也不走欧美奇幻概念图路线。
- 画面应具备自然观察感，重视季节空气、湿度、温度、光线与地表状态，而不是依赖夸张特效制造气氛。
- 叙事强度保持“静中有动”，优先表现节气到来时的环境变化，不做戏剧化剧情插图。
- 单张图应能脱离文案成立，但不能脱离节气语义；观者应能一眼感知季节状态，而不是只看到泛自然风景。

### 9A.2 统一画面纪律

所有节气插画默认遵守以下纪律：

1. 主体清晰，环境辅助，避免信息平均铺满全图。
2. 画面必须能明确拆分为前景 / 中景 / 背景三层，至少两层清楚成立。
3. 主体不超过画面宽度的 `45% - 55%`，避免过大导致嵌入页面后压迫文字。
4. 单张图的主叙事元素不超过 3 组，避免画面主题发散。
5. 色彩以中低饱和为基础，仅在局部使用高亮点缀，不使用整张高纯度撞色。
6. 光感应自然、可感知，但不做强舞台光、霓虹光、神性光束等非节气语境表达。
7. 所有季节变化应通过材质与空气表现出来，例如嫩芽、潮气、积雪、枯枝、土壤、薄雾、水光，而不是只靠滤镜换色。

### 9A.3 统一用色原则

#### 春季样本通用原则
- 主色倾向：灰青、嫩绿、浅土、雾白、淡水色。
- 不使用荧光绿、塑料粉、糖果蓝等人工感过强的颜色。
- 春季不是“鲜艳花海海报”，而是“气候刚刚转松”的层次感。

#### 冷湿天气通用原则
- 雨雾场景优先用灰蓝、灰绿、低对比墨色组织空间。
- 明度可以偏高，但局部对比必须保留，否则画面会发灰发脏。

#### 冬季样本通用原则
- 雪景不等于纯白铺满；必须保留冷灰、蓝灰、树皮深色、屋檐积雪边缘等层次。
- 冬季冷感来自空气和材质，不来自大面积蓝色滤镜。

### 9A.4 统一主体策略

建议主体优先级如下：

1. 季节性自然主体：枝芽、柳枝、草木、雪枝、地表、屋舍、田埂、水岸
2. 节气行为主体：行路、观雨、迎春、候雪、停驻、眺望
3. 点状生命主体：鸟、燕、行人、农人、小童等

约束如下：

- 人物如出现，应服务节气，不应喧宾夺主成为人物海报。
- 动植物应具有可辨识的季候特征，不使用与节气不匹配的泛化素材。
- 同一张图只保留一个主主体，其他元素作为辅助叙事或气氛托底。

### 9A.5 统一材质与细节密度

- 细节密度以“近处可见、远处可读”为准，不做全图同等锐利。
- 前景允许更丰富的枝叶、草、雪粒、水痕、花苞等细节；背景应适度简化。
- 不追求照片级纹理，不追求卡通化勾线，应保持插画感与真实季候观察之间的平衡。
- 画面边缘细节密度应明显低于中心主叙事区，给页面嵌入留出呼吸空间。

### 9A.6 统一非目标风格

以下风格默认不采用：

- 过度国潮海报化
- 影视概念海报式强戏剧打光
- 二次元角色主导构图
- 写实摄影拼贴感
- 扁平儿童绘本风
- 高饱和商业宣传画风
- 赛博、梦幻、魔法、超现实奇观式表达

### 9A.7 页面嵌入一致性要求

所有图在生成时都应默认面向 `solar-term-panel__painting` 使用场景，因此必须满足：

- 左侧独立观看成立，缩小后主体仍清楚。
- 边缘环境可被轻裁，但主主体、主动作、主季候信号不可被裁掉。
- 图像气质不能强到压制右侧标题与正文。
- 画面最好在左侧形成稳定视觉锚点，而不是依赖横向大场景才成立。

## 9B. 代表节气执行样本卡

以下 3 张样本卡不再停留在概念描述层，而是作为可直接进入 AI 成图、人工筛选与前端落位验证的执行型规格。

### 9B.1 立春执行样本卡

#### 节气定位
立春样本负责定义“生发型画面”的基准：不是盛春繁花，而是寒意未退、万物初动、气候刚刚转松的瞬间。

#### 目标气质
- 核心关键词：初醒、微冷、轻亮、含蓄生长、空气转软
- 情绪目标：有希望，但不过分热烈；有生机，但不能像暮春或初夏
- 画面强度：中低强度，偏安静，允许轻微风感

#### 主体建议
优先采用以下主体组合之一：

- 主方案 A：一株刚萌芽的树或枝条作为主主体，枝头出现少量嫩芽，画面中可带一位停步观春的人物小体量剪影
- 主方案 B：近处嫩枝与地表初融作为主体，中景出现低矮田埂或村道，不强制出现人物
- 主方案 C：一只或两只归来的燕子作为点状生命主体，但不可成为唯一主主体，仍需有树枝或地景承接

主体约束：

- 嫩芽必须是“刚出现”，不是大面积花开
- 人物若出现，尺寸建议控制在画面高度的 `18% - 28%`
- 不要用大面积花海代替“立春”

#### 前景 / 中景 / 背景
- 前景：
  - 可用近处枝条、初融湿土、零星草芽、薄薄残雪边缘
  - 细节应轻，不宜过密遮挡主体
- 中景：
  - 主树、主枝、田埂、小路、人物或燕子活动区
  - 中景必须承担主要叙事
- 背景：
  - 淡远山、浅灰天空、稀薄晨雾、未完全苏醒的村落轮廓
  - 背景不应比中景更抢眼

#### Safe Area 要求
- 主体视觉中心放在画面垂直中段略偏上位置，约落在整体高度 `42% - 58%`
- 顶部至少保留 `10%` 空气层，用于天空、薄雾、枝梢延展，不放唯一关键信息
- 底部至少保留 `12% - 15%`，用于湿土、草芽、残雪过渡，不让人物脚部或地表关键纹理贴底
- 左右边缘各自约 `8% - 10%` 范围仅放辅助枝叶、空气、风感，不放人物面部、主芽点、主燕群
- 若采用偏左主树构图，右侧必须留出足够的空场作为视觉回弹，不得两侧同时拥堵

#### Production Notes
- 光线建议为早晨或薄云散光，避免正午高亮
- 立春应保留一点冬末冷意，可有薄残雪、湿地、灰枝，不可直接进入“满园春色”
- 嫩绿只能作为局部信号色，不能铺满整张图
- 若出现人物，服饰应低调、中性色，服务节气环境，不做角色设计感强化
- 允许轻微风感，通过枝梢方向、衣摆、燕子动势表现，但不要出现夸张风暴效果
- 推荐优先测试“单树 / 小路 / 微芽 / 远雾”的稳定组合

#### Negative Constraints
- 禁止大面积盛开桃花、樱花、油菜花海
- 禁止鲜艳到接近海报风的亮绿、亮粉、亮黄
- 禁止主主体贴边、顶边或压到底边
- 禁止晴空通透到近似初夏旅游宣传照
- 禁止人物过大、过精致、过时装化
- 禁止只画抽象嫩叶而缺少地景或空气层，导致季节语义不稳

#### 页面嵌入预期
- 嵌入 `solar-term-panel__painting` 后，观者首先应读到“早春开始了”，而不是“花开了”
- 左侧图像在缩小后，主树或主枝仍应清楚成立，人物如被弱化也不影响节气判断
- 即便裁去部分外围枝条，嫩芽、湿地、冷暖交替空气感仍须保留
- 与右侧文案并置时，画面应形成柔和开篇感，而不是高强度视觉抢占

### 9B.2 清明执行样本卡

#### 节气定位
清明样本负责定义“情绪型画面”的基准：不是阴郁压抑，也不是晴朗踏青海报，而是湿润、明净、含蓄、可感知行旅与追思气息的春日烟雨场景。

#### 目标气质
- 核心关键词：烟雨、轻湿、明净、静行、柳色、水气
- 情绪目标：克制的感伤，干净的湿润感，行走中的安静
- 画面强度：低到中，重点在氛围持续，不在单点爆发

#### 主体建议
优先采用以下主体组合之一：

- 主方案 A：一位撑伞行人沿水边小路缓行，柳枝与微雨作为主要环境信号
- 主方案 B：近处垂柳、中景石桥或小径、远处淡屋舍，人物仅作点景
- 主方案 C：无明确人物也可成立，但必须有“路 / 桥 / 水 / 柳 / 雨”中的至少三项共同支撑，否则清明辨识度不足

主体约束：

- 伞可以作为识别点，但不能成为画面唯一亮点
- 柳枝应柔而有方向，不应画成盛夏浓荫
- 水面可以存在，但不做大面积镜面炫技

#### 前景 / 中景 / 背景
- 前景：
  - 低垂柳枝、细雨、水边草叶、湿石、伞沿局部
  - 前景可轻度遮挡，帮助形成烟雨层次
- 中景：
  - 主行人、小径、石桥、水岸、坡地或村口
  - 中景承担“行旅感”与“清明识别”
- 背景：
  - 雾中屋舍、浅山、朦胧天色、远树
  - 背景应弱化边界，保持空气透度

#### Safe Area 要求
- 主体人物或主路径应位于画面中段偏下到中段位置，约落在高度 `48% - 65%`
- 顶部留白建议达到 `12%` 左右，用于雨幕、雾层、天空，不放唯一识别物
- 底部留白至少 `12%`，让路面、水边、湿地有呼吸，不让伞尖、脚步、桥脚贴底
- 左右边缘不放主伞、人物头部、桥拱核心、柳枝主垂线；边缘可用于雨丝、雾气、次要枝条
- 若采用斜向行路构图，路的消失线需指向中部，不可直接把人物导向边界外

#### Production Notes
- 雨应表现为“可感知的湿”，不是暴雨线条特效
- 清明的明度可以偏高，但对比要柔，不可做暗黑阴雨戏剧风
- 柳色应偏灰绿、嫩绿，不可浓绿压满画面
- 若有人物，服装颜色建议低饱和灰蓝、米白、黛青、浅褐，不要红伞、亮黄衣这类强视觉中心
- 水面可有轻微涟漪与反光，但反光面积不宜成为主视觉
- 画面必须同时成立“雨”和“清”，即湿而不脏、雾而不糊

#### Negative Constraints
- 禁止暴雨、雷电、强烈乌云压城式戏剧天气
- 禁止旅游宣传照式蓝天绿柳高饱和晴景
- 禁止红伞、艳服、鲜花堆叠造成节日海报感
- 禁止桥、塔、亭等建筑主导全画面，挤压清明的空气与湿度表达
- 禁止人物撑伞特写过近，导致画面变成人物情绪海报
- 禁止纯空景且缺乏路、水、柳、雨组合，导致节气辨识度下降

#### 页面嵌入预期
- 嵌入 `solar-term-panel__painting` 后，应先读到“烟雨中的清明”，再读到具体人物或建筑
- 图像缩小后，柳线、水岸与行路方向仍然清楚，不能只剩一团灰雾
- 即使裁掉部分外围雨幕或柳梢，中景的小路、人物、石桥或水岸关系仍应成立
- 与文字并置时，应形成安静、略带追思的呼吸感，不应变成阴沉压迫块面

#### 命名示例

```text
solar-term-qingming-illustration-candidate-v3.png
solar-term-qingming-illustration-final-v1.png
```

### 9B.3 大雪执行样本卡

#### 节气定位
大雪样本负责定义“极端季候型画面”的基准：强调覆盖、沉静、厚雪与寒气，不追求暴风雪灾难感，而是呈现冬季深处的安静重量。

#### 目标气质
- 核心关键词：厚雪、冷寂、覆盖、静止、深冬、低声量
- 情绪目标：安静但不空洞，寒冷但不惨烈，厚重但不压黑
- 画面强度：中等偏低，以环境力量为主，不以动作冲突取胜

#### 主体建议
优先采用以下主体组合之一：

- 主方案 A：被厚雪压覆的树与屋檐，近处雪枝清楚，中景屋舍或院落成立
- 主方案 B：雪中小径或村道，中景一位裹衣行人缓行，人物仅作尺度参照
- 主方案 C：一棵雪树或一组雪竹作为主体，配合低屋、矮墙、远山形成空间层次

主体约束：

- 雪必须有“厚度”和“附着关系”，不能只是白色覆盖滤镜
- 主体宜稳重，不宜轻飘散乱
- 若有人物，人物不超过画面高度的 `16% - 24%`，只作为气候尺度参照

#### 前景 / 中景 / 背景
- 前景：
  - 雪枝、积雪石块、矮墙、雪堆边缘、地表积雪起伏
  - 前景负责建立雪量与材质厚度
- 中景：
  - 主树、屋舍、村道、人物、院门、桥、小坡
  - 中景必须承担主体稳定性
- 背景：
  - 冷灰天空、远山、雪雾、被雪软化的林线
  - 背景不可彻底消失，需保留空间深度

#### Safe Area 要求
- 主主体建议落在画面中部略偏下，约在高度 `50% - 68%` 区间，以获得“雪压下来”的稳定感
- 顶部至少保留 `10% - 12%` 空间给天空、落雪或树梢延展，避免树冠、屋顶雪帽贴边
- 底部至少保留 `14% - 15%`，确保地面积雪起伏、道路边缘、脚印或雪堆不过于拥挤
- 左右边缘只放雪枝延展、雾气、围墙边角等次要信息，不放主屋檐、主树干、人物头部或唯一亮部
- 雪景留白必须是“有结构的留白”，边缘不可空到失去承托，也不可满到没有呼吸

#### Production Notes
- 雪的层次至少通过三种明度区分：高光雪面、冷灰阴影、深色承重物体
- 建议有少量深色结构物帮助画面站稳，例如树干、屋檐、木门、石墙
- 可有轻雪或静雪，但不建议暴雪横扫式运动线
- 冷色可以主导，但局部允许木色、土色、暖窗纸色作为非常克制的平衡点
- 若出现脚印，数量宜少，作为叙事点，不作为图案铺陈
- 推荐优先测试“雪枝 + 屋檐 + 小径”组合，最利于页面嵌入后识别

#### Negative Constraints
- 禁止把整张图做成单一高白度无层次白片
- 禁止蓝色滤镜过重，导致雪景失真
- 禁止暴风雪、灾难片、极寒求生式戏剧叙事
- 禁止屋舍或人物过大，削弱“大雪覆盖环境”的主体关系
- 禁止圣诞装饰、灯串、节庆视觉误导节气语义
- 禁止仅有雪花特效而缺少厚雪附着关系

#### 页面嵌入预期
- 嵌入 `solar-term-panel__painting` 后，应先感知“厚雪压境”，再感知具体屋舍、树木或人物
- 图像缩小后，主树或主屋檐的雪压形态仍须清楚，不能只剩白色块面
- 即便裁掉少量外围雪枝，画面中“雪量、寒意、安静重量”仍应稳定存在
- 与右侧文案并置时，应形成冬季停顿感与沉静感，而不是视觉上过亮或过空

## 10. 文件命名与交付规范

为了后续批量生成、筛选、回填和版本对比，必须从一开始统一文件管理规则。

## 10.1 命名原则

命名应满足：

- 可读
- 可排序
- 可区分节气
- 可区分版本
- 可区分候选与最终

不建议使用纯中文文件名，也不建议使用无语义的随机名作为正式交付名。

## 10.2 推荐命名模式

建议正式资源命名模式如下：

```text
solar-term-{slug}-illustration-v{n}.{ext}
```

示例：

```text
solar-term-lichun-illustration-v1.png
solar-term-qingming-illustration-v2.png
solar-term-daxue-illustration-v1.png
```

如需区分候选稿与最终稿，可扩展为：

```text
solar-term-{slug}-illustration-candidate-v{n}.{ext}
solar-term-{slug}-illustration-final-v{n}.{ext}
```

示例：

```text
solar-term-lichun-illustration-candidate-v3.png
solar-term-lichun-illustration-final-v1.png
```

## 10.3 slug 建议

建议采用稳定的拼音或英文 transliteration，不建议中英混用。示例：

| 节气 | slug |
|---|---|
| 立春 | `lichun` |
| 清明 | `qingming` |
| 大雪 | `daxue` |

后续 24 节气应统一使用同一套命名风格。

## 10.4 文件格式建议

建议交付两类格式：

1. 预览与接入格式：`png`
   - 适合透明度不重要、需要稳定视觉表现的场景
   - 适合作为页面接入首选格式

2. 源成图保留格式：`png` 或高质量 `jpg`
   - 若 AI 输出天然为 JPG，可保留原稿
   - 最终接入前建议统一导出稳定格式

若后续进行前端性能优化，可再评估 WebP/AVIF，但不作为本阶段生产规范主要求。

## 10.5 版本管理建议

建议至少保留以下层级：

- 候选稿：用于比选
- 入选稿：用于页面测试
- 最终稿：用于正式接入

版本号应递增，不覆盖旧版本，避免后续回溯困难。

## 10.6 交付内容清单

每个节气的交付建议至少包含：

- 最终图 1 张
- 候选图若干张（建议保留 2-4 张）
- 对应 `imageBrief`
- 对应 `imageSpec`
- 简短筛选结论（为什么选中这张）

这样能保证后续调整时有据可查。

## 11. 实现建议与风险

## 11.1 实现建议

### 建议一：先做规格稳定，再做批量生成

不要先把 24 张全部生成再回头统一标准。更合理的顺序是：

1. 先用立春、清明、大雪建立规格样本
2. 验证视觉统一性与页面适配性
3. 再扩展到全部 24 节气

### 建议二：预览页采用“规则 + 样本”双层结构

若只有规则，没有样本，执行时容易抽象；若只有样本，没有规则，后续难以复制。两者必须同时出现。

### 建议三：将页面适配前置到出图阶段

不要把“插入 `solar-term-panel__painting` 后再看能不能用”作为主要策略。应在生成阶段就约束构图与安全区。

### 建议四：让 `imageSpec` 尽可能结构化

结构化信息更利于：

- 程序读取
- 模板化 prompt 生成
- 验收自动化
- 批量生成管理

### 建议五：每个样本都写清楚禁忌项

相比“希望画成什么样”，很多时候“不能画成什么样”更能提升成图稳定性。例如：

- 主体贴边
- 元素过满
- 情绪过度戏剧化
- 颜色失真
- 风格过于卡通化或照片化

## 11.2 主要风险

### 风险一：`imageBrief` 过于概念化，导致执行歧义

如果只有诗意描述，没有构图与边界说明，AI 成图结果会分散，且难以稳定复现。

应对方式：

- 用 `imageSpec` 补足硬规则
- 在样本页中把禁忌项写明

### 风险二：画面好看但不适配页面

有些图单看成立，但放入左侧画境区后主体被裁、边缘过挤、视觉失衡。

应对方式：

- 强化 Safe Area
- 在预览页里明确“页面嵌入预期”
- 优先筛选“可嵌入”的图，而不是只筛选“单张最好看”的图

### 风险三：三张样本风格不统一

如果立春、清明、大雪分别走不同画风，后续扩展到 24 节气时会失去整体性。

应对方式：

- 在预览页中明确整体风格基线
- 让样本共享统一的构图纪律与气质边界

### 风险四：后续字段膨胀

如果执行信息没有层次，可能把 `imageBrief` 填得越来越重，最后既不利于阅读也不利于程序处理。

应对方式：

- 明确 `imageBrief` 只管概念
- 明确 `imageSpec` 只管执行
- 避免同类信息重复出现在两个字段中

### 风险五：交付资源命名与版本混乱

没有统一命名时，后续人工挑图、替换图、回滚图都会很困难。

应对方式：

- 一开始就固定命名模式
- 保留版本号与候选/最终状态标识

## 12. 验收标准

本阶段验收不以“是否已经生成全部 24 张插画”为标准，而以“规格是否可直接用于下一阶段生产”为标准。

## 12.1 文档层验收

以下条件同时满足，视为文档达标：

- 明确说明为什么新增独立预览页
- 明确预览页不改动当前首页长卷结构
- 明确只展示立春、清明、大雪 3 个代表节气
- 明确图片主要用于 AI 生成成图
- 明确主规格为 4:5，推荐尺寸为 1536 × 1920
- 明确 Safe Area 规则，包括：
  - 主体位于中部
  - 顶部留 10% - 12%
  - 底部留 12% - 15%
  - 左右避免关键主体贴边
- 明确未来插画插入 `SolarTermPanel` 左侧 `solar-term-panel__painting`
- 明确保留 `imageBrief`，并新增 `imageSpec` 作为执行层字段
- 明确预览页既是视觉预览，也是生产说明页

## 12.2 结构层验收

预览页的说明结构应包含至少以下内容：

- 背景与目标
- 范围与非目标
- 预览页信息架构
- 图片规格
- Safe Area 规则
- 插入位置设计
- 3 个代表节气选择理由
- 数据模型建议
- 预览页内容模块
- 文件命名与交付规范
- 实现建议与风险
- 验收标准

缺失任一核心章节，均视为未达标。

## 12.3 可执行性验收

以下问题都能直接从文档中得到明确答案，视为具备可执行性：

- 图片按什么比例生成？
- 推荐输出多大尺寸？
- 主体应该放在哪里？
- 哪些区域必须留白？
- 为什么只先做 3 个节气？
- 这 3 个节气为什么是立春、清明、大雪？
- 后续图片最终插入页面哪里？
- `imageBrief` 和 `imageSpec` 分别负责什么？
- 文件应该如何命名？
- 最终如何判断一张图是否合格？

## 12.4 进入下一阶段的判定标准

若本规格文档已经能支持以下动作，则可进入下一阶段：

1. 为 3 个代表节气分别编写执行型插画说明
2. 基于统一规格进行 AI 成图
3. 对候选图进行结构化筛选
4. 将入选图回填到 `solar-term-panel__painting` 进行页面验证
5. 在验证成功后推广到全部 24 节气

---

## 13. 结论

本阶段的关键，不是立即生成更多图片，而是先建立一套能长期复用的插画规格体系。

通过独立预览页承载“视觉预览 + 生产说明”的双重角色，可以在不打断现有首页长卷结构的前提下，把以下事情一次性定义清楚：

- 节气插画该如何被描述
- 节气插画该如何被生成
- 节气插画该如何被验收
- 节气插画最终该如何进入页面

选择立春、清明、大雪作为首批 3 个代表节气，是一种高覆盖率、低成本、便于验证的方法。保留 `imageBrief`、新增 `imageSpec`，则能保证创意层与执行层各司其职，避免后续批量化时失控。

因此，本草稿建议作为“二十四节气插画生成阶段”的首版设计规格基线使用，并在 3 个代表样本验证通过后，扩展到完整的 24 节气插画生产流程。
